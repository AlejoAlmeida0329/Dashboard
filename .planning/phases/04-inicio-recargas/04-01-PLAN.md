    Create new pure module. Imports limited to `date-fns` (subDays, differenceInCalendarDays) and type-only `DashboardFilters` from `@/lib/url-state`. NO `lib/format` import (avoid circular — `period.ts` is more foundational); inline the Bogotá ISO timestamp construction with the literal `T00:00:00-05:00` offset (same convention as `bonos.ts:53-62`).

    Export TWO functions:

    ```ts
    /**
     * Given the active filters, return the immediately-prior window of the
     * SAME LENGTH IN DAYS (Bogotá calendar). Returns null when either
     * `from` or `to` is missing — there is no defined "prior" for an
     * unbounded window.
     *
     * Per 04-CONTEXT.md spec:
     *   - "abril 2026" (30d) → "marzo 1–30 2026" (30d), NOT March 1–31.
     *   - 5-day custom range → 5-day window immediately before `from`.
     *
     * Pure: same input → same output.
     */
    export function computePriorPeriod(
      filters: DashboardFilters,
    ): { from: string; to: string } | null;
    ```

    Implementation per 04-RESEARCH.md "Code Examples → Prior-period computation" (literal code, lines 482-523 of RESEARCH):
    - Guard: if `!filters.from || !filters.to` → return null
    - Validate `^\d{4}-\d{2}-\d{2}$` shape on both (defensive, url-state already enforces)
    - Construct `fromDate` and `toDate` with `T00:00:00-05:00` literal offset
    - Guard: `toDate.getTime() < fromDate.getTime()` → return null (malformed)
    - `lengthDays = differenceInCalendarDays(toDate, fromDate) + 1` (inclusive both ends — same convention as `filterBonos`)
    - `priorTo = subDays(fromDate, 1)`
    - `priorFrom = subDays(priorTo, lengthDays - 1)`
    - Return `{ from, to }` formatted as `YYYY-MM-DD` strings (use `formatInTimeZone(d, "America/Bogota", "yyyy-MM-dd")` from `date-fns-tz` — period.ts may import this directly since it doesn't need the full format.ts façade)

    Then export:

    ```ts
    /**
     * Zero-safe percent change. Returns null when prior is 0, NaN, or
     * non-finite — caller renders em-dash. Mirrors the `pctDelTotal`
     * pattern from `bonos.ts:286`.
     *
     * Returns a fraction (0.123 = +12.3%); consumers feed this to
     * `formatPercent` in `lib/format.ts:78`.
     */
    export function pctChange(current: number, prior: number): number | null {
      if (!Number.isFinite(prior) || prior === 0) return null;
      return (current - prior) / prior;
    }
    ```

    Add JSDoc at file top mirroring `bonos.ts` shape (multi-line block explaining purity invariant + non-imports list).

    ```ts
    export interface InicioSummary {
      gmv: number;
      comision: number;
      takeRate: number;        // fraction 0..1; comision/gmv, zero-safe
      empresasActivas: number; // distinct empresa_id count
      bonosVendidos: number;   // count of BONUS+in+completed
    }

    export interface InicioDeltaSummary {
      current: InicioSummary;
      prior: InicioSummary | null;
    }

    export interface GMVPoint {
      bucket: string;   // YYYY-MM-DD or YYYY-Www
      value: number;
    }

    export interface ActiveEmpresaPoint {
      bucket: string;
      count: number;
    }
    ```

    Helper (NOT exported, file-private):
    ```ts
    function startOfDayBogotaTimestamp(yyyymmdd: string): number;
    function endOfDayBogotaTimestamp(yyyymmdd: string): number;
    ```
    (Verbatim copy of the same helpers from `bonos.ts:53-74`. Inline duplication is intentional per Plan 03-02 precedent — DRY across domain modules costs more than the inline ~22 lines.)

    Exports (5 functions):

    1. **`filterCompletedIn(transactions: Transaction[], filters: DashboardFilters): Transaction[]`**
       - Default filter contract: `direction === "in" && status === "completed"` PLUS Bogotá-anchored from/to PLUS optional empresa.
       - Same date-handling convention as `filterBonos` in `bonos.ts:97-138`. Difference: NO `tipo === "BONUS"` filter — Inicio aggregates over ALL completed inflows, not just bonuses (CONTEXT.md vision: GMV is total volume, not just bonos).
       - Empresa filter (when set) matches `t.empresa_id === filters.empresa` AND ignores the literal `__all__` sentinel from `url-state.ts`.

    2. **`summarizeInicio(transactions: Transaction[]): InicioSummary`**
       - Single-pass reduce over `transactions` (assume already filterCompletedIn-ed).
       - `gmv = sum(t.monto)`
       - `comision = sum(t.comision)`
       - `takeRate = gmv > 0 ? comision / gmv : 0` (zero-safe; do NOT divide by 0)
       - `empresasActivas = new Set(transactions.map(t => t.empresa_id)).size`
       - `bonosVendidos = transactions.filter(t => t.tipo === "BONUS").length`
       - Empty-input safe: returns `{ gmv: 0, comision: 0, takeRate: 0, empresasActivas: 0, bonosVendidos: 0 }`.

    3. **`aggregateGMVByDate(transactions: Transaction[]): GMVPoint[]`**
       - Group by `formatInTimeZone(t.fecha, "America/Bogota", "yyyy-MM-dd")` bucket; sum `monto` per bucket.
       - Return sorted ascending by bucket (string sort works for ISO dates).
       - Do NOT zero-fill missing days (same decision as `aggregateBonosByDate` per Plan 02-02 STATE entry).

    4. **`aggregateGMVByWeek(transactions: Transaction[]): GMVPoint[]`**
       - Same as above but bucket = `formatInTimeZone(t.fecha, "America/Bogota", "RRRR-'W'II")` (ISO week-numbering year + ISO week-of-year, e.g. `2026-W18`).
       - Sort ascending by bucket string.

    5. **`aggregateActiveEmpresasByDate(transactions: Transaction[]): ActiveEmpresaPoint[]` and `aggregateActiveEmpresasByWeek(transactions: Transaction[]): ActiveEmpresaPoint[]`**
       - Build `Map<bucket, Set<empresa_id>>`, then emit `{ bucket, count: set.size }`.
       - Same bucket key convention as the GMV functions.
       - Pitfall 11 (RESEARCH.md): A single empresa with 10 tx in one week MUST count as 1 active, not 10.

    Use `dynamic = "force-dynamic"` is NOT relevant here — this is a pure module, no Next.js semantics.

    Add JSDoc on each exported function: short purpose + edge cases (empty input, zero-safe).
