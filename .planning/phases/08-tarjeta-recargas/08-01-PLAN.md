    Define and export 3 types at the top:
    ```ts
    export interface PurchaseSummary {
      totalCompras: number;       // count of completed PURCHASE direction=out rows
      volumenCOP: number;         // sum of Math.abs(transactionAmount) over the same rows
      ticketPromedio: number;     // volumenCOP / totalCompras (0 when totalCompras===0)
    }

    export interface PurchaseAdoption {
      usersWithPurchase: number;  // distinct transferTikintag in purchaseRows
      totalUsers: number;         // distinct transferTikintag in the broader allTx pool
      adoptionRate: number;       // 0..1 ratio (0 when totalUsers===0)
    }

    export interface PurchaseByDate {
      date: string;               // ISO date YYYY-MM-DD
      compras: number;            // count
      volumenCOP: number;         // sum Math.abs(transactionAmount)
    }

    export interface TopCardUser {
      tikintag: string;
      empresa: string | undefined; // best-effort empresa label from first row's transferEmpresa
      compras: number;
      volumenCOP: number;
      ticketPromedio: number;
    }
    ```

    Implement `filterPurchases(transactions: Transaction[], filters: DashboardFilters): Transaction[]`:
    - Tipo guard: `tx.tipo === "PURCHASE"` AND `tx.direction === "out"` (PURCHASE direction=out
      is the canonical "user spends money on card" semantic — matches CARD-V2-01).
    - Period filter: same pattern as `filterRecargas` in `recargas.ts` (use
      `filters.period.from / .to` ISO strings, compare against `tx.date.toISOString().slice(0,10)`).
    - Status filter: when `filters.status` is non-empty, gate on it. When empty, default to
      `["completed"]` (matches CROSS-V2-01 + Phase 6 contract).
    - Empresa filter (cliente-foco): when `filters.empresa` set, keep only rows where
      `tx.transferEmpresa === filters.empresa` (mirror existing recargas.ts pattern).
    - Tipo filter (`filters.tipo`): IGNORED inside this filter — Uso Tarjeta is PURCHASE-by-definition,
      same convention as `filterBonosV2` / `filterRecargas`. Document this in inline JSDoc.

    Implement `summarizePurchases(rows: Transaction[]): PurchaseSummary`:
    - `totalCompras = rows.length`
    - `volumenCOP = sum of Math.abs(row.transactionAmount)`
    - `ticketPromedio = totalCompras > 0 ? volumenCOP / totalCompras : 0`
    - Zero-safe (handle empty rows array → all zeros, no NaN).

    Why not write tests yet: matches Phase 6/7 v2 domain libraries (no test framework wired —
    tsc + lint + build are the verification floor; UI exercises the path).

  <name>Task 2: Add 3 aggregations — aggregatePurchaseAdoption + aggregatePurchasesByDate + aggregateTopCardUsers</name>

    Append below the Task 1 code in the same file.

    Implement `aggregatePurchaseAdoption(allTx: Transaction[], purchaseRows: Transaction[]): PurchaseAdoption`:
    - `usersWithPurchase = new Set(purchaseRows.map(t => t.transferTikintag).filter(Boolean)).size`
    - `totalUsers = new Set(allTx.map(t => t.transferTikintag).filter(Boolean)).size`
    - `adoptionRate = totalUsers > 0 ? usersWithPurchase / totalUsers : 0`
    - Why two inputs (not one): PRD CARD-V2-04 defines adoption as "% usuarios con ≥1 PURCHASE".
      Numerator must come from PURCHASE rows; denominator must come from the broader user pool
      (any transaction in scope). Page composition runs `filterPurchases` once and passes
      `(allTx, purchaseRows)` here so we count distinct tikintags consistently across both.
      Period scoping: use the SAME period-filtered `allTx` the page already computes for any
      other tab — DON'T re-filter inside this fn. The page is the budget-keeper.
    - JSDoc: cite CARD-V2-04 + the PRD baseline (40/235 = 17%) as the expected order of magnitude;
      flag that the live ratio depends on the active period filter.

    Implement `aggregatePurchasesByDate(rows: Transaction[]): PurchaseByDate[]`:
    - Mirror `aggregateRecargasByDate` in recargas.ts: group by `tx.date.toISOString().slice(0,10)`,
      accumulate `compras` count + `volumenCOP` sum.
    - Sort by date ASC.
    - Empty input → `[]`.
    - Zero-fill GAPS NOT required at this layer (chart leaf in Plan 08-02 handles its own empty
      state — same convention as v1 `aggregateRecargasByDate`).
    - Why this signature (no granularity arg): CARD-V2-05 says "tendencia temporal (granularidad
      día/semana/mes)" — the granularity-aware re-aggregation is a UI/leaf concern (same as
      Inicio v1's TimelineChart bucketing). Domain emits daily; chart re-buckets.

    Implement `aggregateTopCardUsers(rows: Transaction[], limit: number = 10): TopCardUser[]`:
    - Group by `tx.transferTikintag` (skip rows where it's `undefined` — defensive).
    - Per group accumulate: count, sum Math.abs(amount), capture first-row's `transferEmpresa`
      as the display label (best-effort — not all PURCHASE rows have empresa populated, so
      `empresa: string | undefined`).
    - Compute per-row ticketPromedio (volumenCOP / compras).
    - Sort by `volumenCOP` DESC.
    - Slice to `limit` (default 10 per CARD-V2-06).
    - Empty input → `[]`.

    NO RE-EXPORT of types from other modules. NO defensive any-typing. NO NaN paths.
