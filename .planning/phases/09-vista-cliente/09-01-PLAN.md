**Type 1 — `ClienteSummary` (CLI-V2-02, 5 KPIs cabecera):**
```ts
export interface ClienteSummary {
  /** tikintag this summary belongs to (echoed for debugging / labelling). */
  tikintag: string;
  /** Display name. First-occurrence-wins per `empresa_nombre`; falls back to tikintag. */
  empresa_nombre: string;
  /** Net wallet balance approximation in COP — sum of inflows minus sum of outflows for completed transactions of this tikintag. Negative possible. */
  balance: number;
  /** Date of earliest transaction (any tipo, any direction, any status) for this tikintag. null if none. */
  primeraTx: Date | null;
  /** Date of latest activity-counting transaction (direction=in && status=completed) for this tikintag. null if none. */
  ultimaActividad: Date | null;
  /** Count of ALL transactions (any tipo / direction / status) carrying empresa_id === tikintag. */
  totalTx: number;
  /** "Pocket activo" — true when the tikintag has at least one activity-counting tx in the last 30 calendar days (Bogotá) relative to filters.to or today. */
  pocketActivo: boolean;
}
```

**Function — `findClienteSummary(transactions, tikintag, filters)`:**
- Single pass over `transactions`. Skip rows whose `empresa_id !== tikintag`. (No filter-window narrowing here — `filters` is consumed only to anchor `pocketActivo`.)
- Accumulate: `totalTx++` on every match; `balance += t.monto` only when `t.status === "completed"` (inflows positive, outflows negative — the tx schema preserves sign so net works); track `primeraTx` (min over `t.fecha`); track `ultimaActividad` (max over `t.fecha` for activity-counting rows only).
- `empresa_nombre` = first-occurrence's `t.empresa_nombre || t.empresa_id`.
- `pocketActivo` math: anchor `asOf = filters.to ? new Date(\`${filters.to}T12:00:00-05:00\`) : new Date()`. Window opens at `asOf - 30*24*3600*1000` (30 days back). True when any activity-counting tx for tikintag has `t.fecha.getTime() >= window`.
- Return `null` when `totalTx === 0` (unknown tikintag — caller renders 404 fallback).
- JSDoc with one runnable example showing inputs / output.

**Type 2 — `ClienteBenchmark` (CLI-V2-07):**
```ts
export interface ClienteBenchmark {
  /** Cliente's avg processing time across COMPLETED payouts attributable to tikintag (minutes). 0 when no payouts found. */
  clienteMinutes: number;
  /** Platform-wide avg processing time across all COMPLETED payouts (minutes). 0 when none. */
  platformMinutes: number;
  /** clienteMinutes - platformMinutes. Negative = cliente faster than platform (the value-prop number). */
  deltaMinutes: number;
  /** Number of completed payouts for the cliente (denominator transparency). */
  clienteSampleSize: number;
  /** Number of completed payouts for the platform (denominator transparency). */
  platformSampleSize: number;
}
```

**Function — `aggregateClienteBenchmark(joinedPayouts, tikintag)`:**
- Input: pre-joined `JoinedPayout[]` (page composition runs `joinPayouts(allTx, allPayouts)` ONCE per request and threads it here — same one-JOIN-per-request budget as Plan 07-04).
- Filter to `state === "completed"` defensively (mirrors `aggregateAverageProcessingMinutes` in payouts.ts:582). `latencySeconds` for `completed` rows carries `Total Time`; `Aging` fallback would contaminate the mean.
- Cliente subset: completed payouts where `p.transaction?.empresa_id === tikintag`. Unmatched rows (transaction undefined) cannot be attributed → SKIP.
- Compute: cliente mean (latency_seconds / 60), platform mean (over ALL completed payouts), delta = cliente - platform. Sample sizes carried for transparency.
- All return values default to 0 when sample size is 0; never NaN/Infinity.
- JSDoc cites the negative-delta-as-good convention so UI can render "X min más rápido que el promedio" when `delta < 0`.

**Type 4 — `ClienteP2P`:**
```ts
export interface ClienteP2P {
  /** Count of P2P direction=in completed transactions for tikintag. */
  countIn: number;
  /** Count of P2P direction=out completed transactions for tikintag. */
  countOut: number;
  /** Sum of monto for direction=in (positive). */
  montoIn: number;
  /** Absolute sum of monto for direction=out (positive — Math.abs applied). */
  montoOut: number;
  /** Last 50 P2P rows (any direction, any status), date DESC. Cap prevents pathological renders if a tikintag has thousands of P2Ps. */
  rows: ClienteP2PRow[];
}
```

**Function — `aggregateClienteP2P(transactions, tikintag, filters)`:**
- Filter: `t.tipo === "P2P"` AND `t.empresa_id === tikintag` AND Bogotá-window from filters (start/endOfDayBogotaTimestamp). `filters.status` honored CSV-style with default `["completed"]` for the COUNT/MONTO accumulators (rejected P2Ps don't count towards "actividad real"); but the `rows` array carries ALL P2Ps in window regardless of status (for the table — completed AND rejected are both worth showing in the dossier).
- Counters: only completed rows. `countIn` += 1 when `direction === "in"`, `montoIn += t.monto`. `countOut` += 1 when `direction === "out"`, `montoOut += Math.abs(t.monto)`.
- Rows: collect ALL in-window P2P rows (any status) → sort by `fecha` DESC → `slice(0, 50)`.
- `contraparte` derivation: `direction === "in"` → `t.sourceTransferTikintag`; `direction === "out"` → `t.destinationTransferTikintag`. Carry undefined verbatim when Sheet cell empty (UI renders `—`).
- Empty input → `{ countIn: 0, countOut: 0, montoIn: 0, montoOut: 0, rows: [] }`. Never NaN.
- JSDoc with example showing the in/out split + a row.

**Function — `aggregateClienteTimeline(transactions, joinedPayouts, tikintag, filters)`:**
- Bogotá window from filters; both transactions AND joinedPayouts narrowed to window.
- Source 1 — transactions where `t.empresa_id === tikintag`:
  - tipo=BONUS direction=in → `BONUS_IN`, counterparty = `t.sourceTransferTikintag`.
  - tipo=BONUS direction=out → `BONUS_OUT`, counterparty = `t.destinationTransferTikintag`.
  - tipo=P2P direction=in → `P2P_IN`, counterparty = `t.sourceTransferTikintag`.
  - tipo=P2P direction=out → `P2P_OUT`, counterparty = `t.destinationTransferTikintag`.
  - tipo=PURCHASE direction=out → `PURCHASE`, counterparty = `t.empresa_nombre` (the merchant; for now equals tikintag itself but Phase 2 decision lets schema upgrade transparently).
  - tipo=PAYIN_PSE → `RECHARGE_PSE`, counterparty = `"PSE"`.
  - tipo=PAYIN_TRANSFER → `RECHARGE_TRANSFER`, counterparty = `"Transfer"`.
  - Other tipos / direction=OTRO_DIRECTION → `OTRO` (still surfaced, never silently dropped).
- Source 2 — joinedPayouts where `p.transaction?.empresa_id === tikintag`:
  - Always `PAYOUT_BANK` (the only tipo in BD_Payouts), counterparty = `p.medium` (bank code), status = `p.state` (completed/in_progress/failed).
  - Use `p.transactionId` as id; `p.fecha` as fecha; `p.monto` as monto.
- Merge both arrays → sort by `fecha.getTime()` DESC → `slice(0, 200)` (defensive cap, same magnitude as P2P cap; the UI then renders all).
- monto: `Math.abs(t.monto || 0)` for transactions; `p.monto` for payouts.
- Empty input → `[]`.
- JSDoc with example showing a 3-event mix (BONUS_IN, PURCHASE, PAYOUT_BANK).
