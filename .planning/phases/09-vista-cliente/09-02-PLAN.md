**1) `src/components/clientes/TikintagSelector.tsx`** — Client Component (mirrors `src/components/filters/empresa-filter.tsx` pattern but navigates to `/clientes/[id]` instead of writing `?empresa=…`).

```tsx
"use client";

/**
 * TikintagSelector — section-level dropdown that switches the dossier
 * to a different tikintag without going back to /clientes (CLI-V2-01).
 *
 * Mirrors `EmpresaFilter` in `src/components/filters/` but writes the
 * selection to the URL PATH (/clientes/{id}) rather than to a search-
 * param. Path-based selection preserves the established v1.0 dynamic-
 * route convention (the existing share-URL flow already uses
 * /clientes/[empresaId]).
 *
 * On change: router.push("/clientes/{newId}" + preserved query string).
 * The current searchParams (filters + presenter mode) ride along so the
 * user keeps their date range / presenter context across switches.
 */
```

- Props: `{ options: { id: string; nombre: string }[]; current: string; }`.
- Imports: `usePathname` is NOT used (we hardcode `/clientes/{id}`); use `useRouter` + `useSearchParams` from `next/navigation`. Use `Label` from `@/components/ui/label` (sr-only).
- Render a native `<select>` styled exactly like `EmpresaFilter`'s select className. No "(Todas las empresas)" option — selector always represents ONE current tikintag (no "all" mode in dossier section).
- onChange: `const next = e.target.value; const qs = searchParams.toString(); router.push(qs ? \`/clientes/${encodeURIComponent(next)}?${qs}\` : \`/clientes/${encodeURIComponent(next)}\`);`.
- `encodeURIComponent` is required because tikintags carry `$` (`$mario` → `%24mario` in path segments per RFC 3986).
