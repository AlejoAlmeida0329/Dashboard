       - `volumenOut: number` — sum of `Math.abs(monto)` across rows where `direction === 'out'` (PURCHASE/PAYOUT-out montos are negative in BD_Plataforma — same `Math.abs` convention as `summarizePurchases` in `cardUsage.ts:213`)

       - `successRate: number` — `countCompleted / total`; zero-safe `0` when `total === 0`.

    3. **`export interface TransactionTypeBucket`:**
       - `tipo: string` — the literal `TransactionType` value OR the rolled-up label `"Otros"` for the tail. (Type kept as `string` so `"Otros"` can sit alongside the typed enum values; mirror `BancoStats.medium` pattern in `payouts.ts:278`.)
       - `count: number` — count of rows in this bucket
       - `share: number` — `count / total` (0..1); zero-safe `0` when `total === 0`.

    4. **`export function filterInicioV2(transactions: Transaction[], filters: DashboardFilters): Transaction[]`:**
       - State-UNFILTERED by default (so `summarizeInicioV2` can compute the success rate from one input). When `filters.status` is non-empty, narrow to that Set (same `Set<string>(filters.status)` pattern as `filterPayoutsV2:515-518`).
       - Honor `filters.tipo` CSV: when non-empty, narrow to that Set (mirror the `filters.status` shape; this is INI-V2-04's complement — the donut-by-tipo is over the full type universe by default, but the URL filter narrows everything if present).

       - Honor `filters.empresa` when set (`t.empresa_id === filters.empresa`).
       - Allow BOTH `direction === 'in'` and `direction === 'out'` (drop the v1 `direction !== 'in' return false` check). Skip ONLY `direction === 'OTRO_DIRECTION'` (defensive — keeps the IN/OUT split sums clean).
       - Pure: returns a new array.

    5. **`export function summarizeInicioV2(transactions: Transaction[]): InicioSummaryV2`:**
       - Single-pass reduce (no intermediate arrays). For each row: increment IN/OUT volume per `direction`; increment status counter per `status`; add `tikintag` to a Set.
       - Empty input → all zeros, `successRate: 0`.

    6. **`export function aggregateTransactionTypeDistribution(transactions: Transaction[], n = 6): TransactionTypeBucket[]`:**
       - Group input by `tipo` into `Map<TransactionType, number>` counts.
       - Sort groups DESC by count.
       - Top N → returned as-is. Remaining tail → rolled up as a single `{ tipo: "Otros", count: sum, share: ... }` bucket.
       - When `total === 0` → return `[]` (no point in emitting an "Otros" zero-row).
       - When `groups.length <= n` → no Otros bucket; return all groups.
       - Each bucket's `share = count / total` (computed using the input's full length as the denominator; zero-safe 0).
       - Sort the final array DESC by count (Otros bucket — if present — sits at the end ONLY when its count would not displace any top-N entry; sort by count keeps ordering deterministic; ties broken by `tipo` lexicographic ascending). Pure. Returns a new array.

       - `usuariosActivos: number` — DISTINCT tikintag count in this bucket (mirror v1 `ActiveEmpresaPoint.count` Set-per-bucket convention to avoid Pitfall 11).

    2. **`export function aggregateActivityByDateV2(transactions: Transaction[]): ActivityPointV2[]`:**
       - Map keyed by Bogotá `yyyy-MM-dd` bucket → `{ tikintags: Set<string>, volumen: number }` accumulator.
       - For each row: bucket-derive via `formatInTimeZone(t.fecha, BOGOTA_TZ, "yyyy-MM-dd")`; add `tikintag` to set; sum `monto`.
       - Emit sorted ASCENDING by bucket (string-sortable). NO zero-fill (Recharts handles continuous axis spacing — same convention as v1 `aggregateGMVByDate:259`).
       - Empty input → `[]`. Pure.

    3. **`export function aggregateActivityByWeekV2(transactions: Transaction[]): ActivityPointV2[]`:**
       - Same as ByDateV2 but with bucket format `RRRR-'W'II` (ISO week-numbering year + ISO week-of-year — mirror v1 `aggregateGMVByWeek:283` rationale block).
       - NO zero-fill. Sorted ascending. Empty → `[]`. Pure.

    4. **`export interface TopUserVolumeRow`:**
       - `tikintag: string`
       - `empresa: string | undefined` — first observed `empresa_nombre` for this tikintag (denormalized label for table display; mirror `TopRecharger.empresa` shape from `recargas.ts:371`)
       - `volumenIn: number` — sum of `monto` for `direction === 'in'` rows for this tikintag
       - `volumenOut: number` — sum of `Math.abs(monto)` for `direction === 'out'` rows
       - `volumenNeto: number` — `volumenIn - volumenOut` (this is the ranking key per INI-V2-06 "top usuarios por volumen" — net activity is the operator-relevant headline; positive = net receiver, negative = net spender)
       - `transacciones: number` — count of rows for this tikintag (any direction, any status)

    5. **`export function aggregateTopUsersByVolume(transactions: Transaction[], limit = 10): TopUserVolumeRow[]`:**
       - Group input by `tikintag` into accumulator Map.
       - For each tikintag: aggregate `volumenIn`, `volumenOut`, `transacciones`; capture first-observed `empresa_nombre` (first-occurrence-wins, same convention as `findTopEmpresaByGMV:113-122`).
       - Compute `volumenNeto = volumenIn - volumenOut` per tikintag.
       - Sort DESC by `volumenNeto`; deterministic tiebreak: then DESC by `transacciones`, then `tikintag` lexicographic ASC.
       - Slice to top `limit`. Empty input → `[]`.
       - JSDoc cite INI-V2-06 + reference the v1→v2 user-lens shift (top by tikintag, not by empresa) — same convention catalogued in STATE.md (Plan 08-04).
