Crear módulo PURO (sin imports de Next, sin "server-only") — son funciones que toman `Transaction[]` y devuelven shapes derivados. Esto las hace usables tanto en Server Components como en futuras tests/scripts.

Imports:
  - `import type { Transaction } from "./types";`
  - `import type { DashboardFilters } from "@/lib/url-state";` (solo para el tipo del filtro)

**Constantes:**
- `BONO_TRANSACTION_TYPES`: array readonly de los `TransactionType` que cuentan como "bono". Por default `["BONO"]`. Si la inspección de Plan 01 reveló que los bonos vienen como `BONO_VENTA` o `EMISION_BONO`, ajustar esta constante. Documentar la decisión en jsdoc.
- `MIN_DATE` / `MAX_DATE`: helpers para parsear `filters.from` / `filters.to` (YYYY-MM-DD interpretados en Bogotá). Reusar la lógica de `url-state.ts`/`format.ts` — convertir el string ISO date a un Date que represente las 00:00 / 23:59 en zona Bogotá. Si parseo falla, default a -Infinity / +Infinity.

**Tipos exportados:**
```ts
export interface BonoSummary {
  count: number;            // # de bonos en el rango
  ticketPromedio: number;   // sum(monto) / count, o 0 si count=0
  comisionTotal: number;    // sum(comision) — la métrica revenue
  montoTotal: number;       // sum(monto) — total $ vendido
}

export interface BonoByDate {
  date: string;   // 'YYYY-MM-DD' en Bogotá
  count: number;  // # bonos vendidos ese día
  monto: number;  // $ vendido ese día (suma de Transaction.monto)
}

export interface BonoByEmpresa {
  empresa_id: string;
  empresa_nombre: string;
  count: number;
  monto: number;       // $ vendido por esa empresa
  comision: number;    // $ comisión generado por esa empresa
  pctDelTotal: number; // monto de la empresa / monto total de TODOS los bonos del rango (fracción 0..1)
}
```

**Funciones exportadas:**

1. `filterBonos(transactions: Transaction[], filters: DashboardFilters): Transaction[]`
   - Aplica en orden: tipo (must be in BONO_TRANSACTION_TYPES), fecha (>= from si presente, <= to si presente), empresa (filters.empresa equals Transaction.empresa_id si presente).
   - Status filter: por DEFAULT, **incluir solo `status === 'SUCCESS'`** (o lo que Plan 01 reveló como "estado completado"). Si Plan 01 no encontró un status claramente "completed", incluir todos los status y documentar.
   - Direction filter: incluir solo `direction === 'in'` para representar ventas (los `out` son reversos). Documentar en jsdoc.

2. `summarizeBonos(bonos: Transaction[]): BonoSummary`
   - Pure: cuenta, suma monto, suma comision, calcula promedio. Si count=0, ticketPromedio=0 (no NaN).

3. `aggregateBonosByDate(bonos: Transaction[]): BonoByDate[]`
   - Agrupa por `formatBogotaDate(t.fecha)` → 'DD/MM/YYYY' formateado, pero para uso en chart usar el ISO 'YYYY-MM-DD' (usar `toBogotaISODate` de `format.ts`). Salida ordenada ascendente por date.
   - Garantiza que NO haya gaps explícitos — si no hay bonos un día, ese día simplemente no aparece (la chart lib del Plan 03 maneja el spacing). La densidad de bonos diarios es alta (3232 transacciones totales → muchos bonos por día), no necesitamos zero-fill.

4. `aggregateBonosByEmpresa(bonos: Transaction[]): BonoByEmpresa[]`
   - Agrupa por `t.empresa_id`. Para `empresa_nombre` usar el primer valor visto (deberían ser idénticos si vienen del mismo origen; en Phase 2 empresa_id == empresa_nombre por la decisión de Plan 01).
   - Calcula `pctDelTotal` con respecto al `montoTotal` agregado de TODOS los bonos pasados a la función (NO al universo de transacciones — el filtro ya pasó por filterBonos antes).
   - Ordenar descendente por `monto`.

5. `top10Empresas(rows: BonoByEmpresa[]): BonoByEmpresa[]`
   - Toma las primeras 10 entradas (asume input ya ordenado por monto desc — typical use: `top10Empresas(aggregateBonosByEmpresa(bonos))`).
   - Si rows.length < 10, devuelve todas.

**Tests inline (jsdoc o comentarios):**
No agregar test framework en Phase 2 (scope creep). Pero documentar en jsdoc de cada función un ejemplo de input/output para que sea trivial verificar mentalmente.

**Pitfalls cerrados:**
- Pitfall: `pctDelTotal` cuando `montoTotal === 0` → return 0, no NaN/Infinity.
- Pitfall: filtros mal parseados (e.g., `from='not-a-date'`) → tratar como sin filtro, no crashear. Usar try/catch alrededor del Date parse.
- Pitfall: Transactions con `status='OTRO_STATUS'` (de schemas.ts coerce fallback) → exclude por seguridad si BONO_TRANSACTION_TYPES filter ya pasó pero status no es success.

**`src/lib/domain/empresas.ts`** (nuevo):
- Imports: `import type { Transaction } from "./types";` y `import type { EmpresaOption } from "@/components/filters/empresa-filter";`
- Export: `getEmpresaRegistry(transactions: Transaction[]): EmpresaOption[]`
  - Itera transactions, construye un Map<empresa_id, empresa_nombre> (primera ocurrencia gana, mismo patrón que `headerIndexMap`).
  - Retorna `Array.from(map).map(([id, nombre]) => ({ id, nombre })).sort((a, b) => a.nombre.localeCompare(b.nombre, 'es'))`.
  - Si transactions.length === 0, retornar [].
- jsdoc: explica que en Phase 2 empresa_id == empresa_nombre (default tikintag) y que cuando exista una columna de display name, este helper la mapeará.

**`src/lib/sheets/transactions.ts`** (modificación):
- Añadir un nuevo export `getCachedTransactions` que envuelva `getTransactions` con React `cache()`:
  ```ts
  import { cache } from "react";
  // ...existing code...
  export const getCachedTransactions = cache(getTransactions);
  ```
- React `cache()` deduplica por argumentos dentro del MISMO request en Server Components — exactamente lo que necesitamos para que DashboardHeader y /bonos compartan una sola llamada.
- jsdoc: explica el contrato (same-request dedup, no cross-request caching, los Server Components downstream deben usar `getCachedTransactions` en vez de `getTransactions` cuando ya pueda haber sido invocada en el mismo render).

**`src/components/layout/dashboard-header.tsx`** (modificación):
- Convertir el componente a `async` (Server Component).
- Importar `getCachedTransactions` de `@/lib/sheets/transactions` y `getEmpresaRegistry` de `@/lib/domain/empresas`.
- Dentro del componente, antes del `return`:
  ```ts
  let empresas: EmpresaOption[] = [];
  try {
    const result = await getCachedTransactions();
    empresas = getEmpresaRegistry(result.rows);
  } catch (err) {
    // Sheet read failed (creds missing, schema mismatch, etc.) — degrade gracefully.
    // EmpresaFilter renderea con [] como en Phase 1. El error real surface en /bonos page.
    console.error("[DashboardHeader] empresa registry failed:", err);
  }
  ```
- Pasar `<EmpresaFilter empresas={empresas} />` en lugar de `empresas={[]}`.
- Nota: `dashboard-header.tsx` ya está en el árbol async del layout (`ProtectedLayout` es async). Convertir DashboardHeader a async no requiere cambios al layout, solo al `import` y la firma.
