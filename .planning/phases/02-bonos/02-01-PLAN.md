**`src/lib/domain/types.ts`:**

- `DestinationType`: mantener `'tarjeta' | 'cuenta_bancaria'`.
- `TransactionDirection`: nuevo, `'in' | 'out'` (de la columna `direction`).
- `TransactionStatus`: nuevo, derivado de `distinctStatuses` de Task 1. Si los valores son e.g. `['SUCCESS', 'PENDING', 'FAILED']` ese es el enum. Mantener `OTRO_STATUS` fallback si hay valores raros.
- Reemplazar `interface Transaction` con la forma completa:
  ```ts
  export interface Transaction {
    /** Stable id from BD_Plataforma.transaction_id (no longer synthetic). */
    id: string;
    /** Parsed timestamp from created_at. */
    fecha: Date;
    /** Numeric amount from BD_Plataforma.amount (COP). */
    monto: number;
    /** Gross amount before fees (BD_Plataforma.gross_amount). */
    grossAmount: number;
    /** Total transaction fee — what Tikin charges (sum of fixed + variable). */
    comision: number;
    /** Fixed transaction fee component. */
    fixedFee: number;
    /** Variable fee % (e.g. 0.025 for 2.5%). */
    variableFeePct: number;
    /** Normalized transaction category. */
    tipo: TransactionType;
    /** Direction in/out. */
    direction: TransactionDirection;
    /** Transaction status (success/pending/failed/etc.). */
    status: TransactionStatus;
    /** DEFAULT empresa identity = tikintag. To switch to account_id, change ONLY transactions.ts → empresa_id mapping. */
    empresa_id: string;
    /** Display name. Currently same as empresa_id since BD_Plataforma has no display column. */
    empresa_nombre: string;
    /** Raw tikintag value (always available). */
    tikintag: string;
    /** Raw account_id value (always available). */
    accountId: string;
    /** Reference / batch_reference from raw row, useful for joins to BD_Payouts. */
    reference?: string;
    /** Present when this row is a payout-shaped transaction. */
    destination_type?: DestinationType;
  }
  ```
- Documentar en jsdoc del archivo: "Phase 2 Plan 01 rewrote this to match real BD_Plataforma headers (23 cols). See 02-01-SUMMARY.md for empresa_id default rationale."

- `TransactionRowSchema`: Zod object con cada header → coercion correcta:
  - `transaction_id`: `z.string().min(1)`
  - `created_at`: `z.coerce.date()` (sigue siendo FORMATTED_STRING)
  - `tikintag`: `z.string().min(1)`
  - `account_id`: `z.string().min(1)`
  - `amount`, `gross_amount`, `fixed_transaction_fee`, `total_transaction_fee`: `z.coerce.number().finite().min(-1e12).max(1e12)` (rango aumentado vs Phase 1 porque amounts en COP pueden ser grandes)
  - `variable_fee_percentage`: `z.coerce.number().finite().min(0).max(1)` (asumiendo fracción 0..1; si Task 1 reveló que viene como entero 0..100, dividir por 100 en el transform)
  - `direction`: `z.string().transform(s => s.trim().toLowerCase())` → coerce a `TransactionDirection` con OTRO fallback
  - `transaction_type`: `z.string().transform(s => s.trim().toUpperCase())` → coerce a `TransactionType` con OTRO fallback (mismo patrón que Phase 1)
  - `status`: `z.string().transform(s => s.trim().toUpperCase())` → coerce a `TransactionStatus` con OTRO_STATUS fallback
  - `reference`, `wallet_id`, `balance_*`, `source_*`, `destination_*`, `batch_reference`, `pocket_name`: `z.string().optional().or(z.coerce.string().optional())` (no críticos para Phase 2; se cargan crudos)
- `.transform((parsed): Transaction => ({...}))` arma el Transaction con:
  - `id: parsed.transaction_id`
  - `fecha: parsed.created_at`
  - `monto: parsed.amount`
  - `grossAmount: parsed.gross_amount`
  - `comision: parsed.total_transaction_fee` (decisión: usar total como "comisión" total que cobra Tikin)
  - `fixedFee: parsed.fixed_transaction_fee`
  - `variableFeePct: parsed.variable_fee_percentage`
  - `tipo: parsed.transaction_type`
  - `direction: parsed.direction`
  - `status: parsed.status`
  - `tikintag: parsed.tikintag`
  - `accountId: parsed.account_id`
  - **`empresa_id: parsed.tikintag`** (DEFAULT — punto único de cambio si el usuario decide account_id)
  - **`empresa_nombre: parsed.tikintag`** (DEFAULT — sin columna de display name disponible)
  - `reference: parsed.reference`
  - `destination_type: undefined` (Phase 2 no la consume; Phase 3 Payouts la deriva de BD_Payouts.Destination Medium)
- Borrar la constante vieja `KNOWN_TRANSACTION_TYPES` y el bloque tentativo. Reemplazar con las nuevas constantes derivadas de Task 1.
