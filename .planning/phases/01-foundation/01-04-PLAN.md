<task type="auto">
  <name>Task 1: Vercel link + env vars set + first deploy</name>
  <files>vercel.json, .vercel/project.json</files>
  <action>
1. **Crear `vercel.json` en raíz**:
   ```json
   {
     "regions": ["iad1"],
     "framework": "nextjs"
   }
   ```
   Notas: Vercel detecta Next.js automáticamente; pin explícito de region asegura proximidad a Google APIs (Pitfall 5/7 / latencia + cold start).

2. **Verificar autenticación Vercel**:
   - Ejecutar `vercel whoami`. Si retorna 'Not authenticated' o error similar → CHECKPOINT dinámico:
     ```
     CHECKPOINT: Action Required — Vercel auth gate
     Run: vercel login
     Type "done" cuando completes el browser auth.
     ```
   - Si responde con email del usuario → continuar.

3. **Vincular directorio al proyecto**:
   - `vercel link --yes` — usa defaults de inferencia (current dir, default scope, project name = directorio name o `tikin-dashboard`). Si pregunta, aceptar nombre `tikin-dashboard`.
   - Verificar: `cat .vercel/project.json` muestra `projectId` y `orgId`.

4. **Configurar env vars en Vercel** (target: production + preview + development).
   Para CADA env var del proyecto, usar `vercel env add <NAME> production` (o `printf '%s' "<VALUE>" | vercel env add <NAME> production --force`):
   - `SESSION_SECRET` → mismo valor que en .env.local (NO regenerar, queremos que sessions hechas en local sigan válidas si abrimos prod, aunque en práctica son ambientes separados; OK regenerar).
   - `DASHBOARD_PASSWORD_HASH` → mismo bcrypt hash que .env.local (mismo password aplicará en prod).
   - `UPSTASH_REDIS_REST_URL` → del setup de Upstash del usuario.
   - `UPSTASH_REDIS_REST_TOKEN` → del setup de Upstash del usuario.
   - `GOOGLE_SERVICE_ACCOUNT_EMAIL` → del setup de GCP del usuario.
   - `GOOGLE_SERVICE_ACCOUNT_PRIVATE_KEY` → **CRITICAL**: pegar con `\n` LITERALES (no newlines reales). Si el usuario provee un .json, extraer el campo `private_key` y reemplazar newlines reales con `\\n` escaped antes de pasar a `vercel env add`. El runtime hace `.replace(/\\n/g, '\n')`.
   - `GOOGLE_SHEETS_TRANSACTIONS_ID` → del setup del usuario.
   - `GOOGLE_SHEETS_PAYOUTS_ID` → del setup del usuario.

   **Si CUALQUIER env var no la tenemos disponible** (porque el usuario aún no completó user_setup), CHECKPOINT dinámico listando exactamente cuáles faltan y los pasos para obtenerlas. NO inventar valores.

5. **Marcar como Sensitive** (por seguridad, Pitfall 1 + Vercel April 2026 incident):
   - `vercel env ls` para ver el estado actual.
   - Para `DASHBOARD_PASSWORD_HASH`, `SESSION_SECRET`, `UPSTASH_REDIS_REST_TOKEN`, `GOOGLE_SERVICE_ACCOUNT_PRIVATE_KEY`: idealmente usar `--sensitive` flag al añadir (Vercel Sensitive Environment Variables). Si el CLI no soporta el flag, hacer una nota en el SUMMARY para que el usuario lo marque en el Vercel dashboard.

6. **Deploy a producción**:
   - `vercel --prod --yes`. Capturar la URL de production de la salida (formato `https://tikin-dashboard-<hash>.vercel.app` o el alias del proyecto).
   - Verificar: `vercel ls` muestra el deployment como READY.

<task type="auto">
  <name>Task 2: Production smoke test — auth + Sheets reads end-to-end</name>
  <files></files>
  <action>
**Verificar la URL de production con curl** (reemplazar `$PROD_URL` con la URL real capturada del Task 1):

1. **Auth redirect (sin cookie)**:
   ```bash
   curl -i -s -o /dev/null -w "%{http_code} %{redirect_url}\n" $PROD_URL/inicio
   ```
   Expected: `307 https://.../login` o similar redirect a /login.

2. **Login page accesible**:
   ```bash
   curl -i -s $PROD_URL/login | grep -E "(password|Tikin|<form)" | head -5
   ```
   Expected: matches del HTML del login form.

3. **Login funcional + cookie set**:
   - Server Actions usan POST con un body multipart o form-urlencoded más un header `Next-Action`. Curl-testing es complejo; alternativa:
     - **Opción A (browser-mediated)**: Mostrar la URL al usuario y dejar el manual login para Plan 04 verification. Pero `skip_checkpoints: true` en config sugiere automatizar.
     - **Opción B (CLI con node)**: Escribir un pequeño script que use `fetch` con FormData para POST al login Server Action. Capturar el `set-cookie` header. Usar esa cookie para los siguientes calls. **Hacer esta opción**:
       ```js
       // scripts/_prod-smoke.mjs (temporal — se borra al final de Task 2)
       const PROD_URL = process.env.PROD_URL;
       const PASSWORD = process.env.DASHBOARD_PASSWORD; // texto plano que matchea el hash
       // ... POST a /login (Server Action), capturar cookie, GET /api/_smoke con cookie
       ```
     - El script escribe un report a stdout: ok/fail por step. Si todo OK, borrar el script (`rm scripts/_prod-smoke.mjs`).

4. **Smoke endpoint con cookie válida**:
   ```bash
   curl -s -H "Cookie: session=<token>" $PROD_URL/api/_smoke
   ```
   Expected: `{"ok": true, "count": N, "lastReadAt": "...", "skipped": M, "sample": [...]}`.

   **Si retorna `{"ok": false, "error": "Sheet schema mismatch — columnas faltantes: ..."}`**: NO es un fallo del deploy — es información valiosa de que los headers reales del Sheet difieren de los tentativos en `src/lib/domain/schemas.ts`. Anotar QUÉ columnas faltan en el SUMMARY como blocker para Phase 2 (Bonos). Plan 04 sigue success — el deploy está bien; el schema necesita ajuste en una phase decimal o early en Phase 2.

5. **App shell visualmente accesible**:
   - `curl -s $PROD_URL/login | head -50` para confirmar que el HTML server-rendered incluye el form.
   - Si estamos en autonomous mode: NO bloquear con checkpoint:human-verify. Reportar la URL al usuario en el output del SUMMARY como "abrir y validar visualmente".

6. **Logging de la URL de prod en el SUMMARY**: anotar la URL exacta para que el usuario tenga el link a mano.
