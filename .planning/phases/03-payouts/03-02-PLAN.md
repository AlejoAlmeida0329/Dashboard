2. `quantileSorted(sortedValues: readonly number[], p: number): number`
   - Implements R-7 linear interpolation (the default in R, NumPy, Excel `PERCENTILE.INC`, `simple-statistics.quantileSorted`).
   - Algorithm:
     ```
     if (sortedValues.length === 0) return 0;          // zero-safe — Pitfall: percentile of empty set
     if (sortedValues.length === 1) return sortedValues[0];
     if (p <= 0) return sortedValues[0];
     if (p >= 1) return sortedValues[sortedValues.length - 1];
     const h = (sortedValues.length - 1) * p;          // R-7 position
     const lo = Math.floor(h);
     const hi = Math.ceil(h);
     if (lo === hi) return sortedValues[lo];
     return sortedValues[lo] + (h - lo) * (sortedValues[hi] - sortedValues[lo]);
     ```
   - INPUT MUST BE SORTED ASCENDING — document this in JSDoc; callers sort once and reuse for p50 + p95.
   - Returns 0 for empty input (zero-safe). Returns the single value for n=1. Linear interpolation otherwise.
   - Document with concrete fixture: `quantileSorted([1, 2, 3, 4, 5], 0.5) === 3` (median exact). `quantileSorted([1, 2, 3, 4], 0.5) === 2.5` (between v[1]=2 and v[2]=3, interpolated).

**1. `aggregateLatencyHistogram(rows: Payout[]): LatencyBucket[]`**

Output type:
```ts
export type LatencyBucketLabel = "<1h" | "1-6h" | "6-24h" | ">24h";

export interface LatencyBucket {
  bucket: LatencyBucketLabel;
  /** Count of payouts where medium === "tarjeta" and latency in this bucket. */
  tarjeta: number;
  /** Count of payouts where medium === "cuenta_bancaria" and latency in this bucket. */
  banco: number;
  /** Count of payouts where medium === "OTRO_MEDIUM" and latency in this bucket. */
  otro: number;
  /** Total count in this bucket (tarjeta + banco + otro) — convenience for tooltip + totals. */
  total: number;
}
```

Bucket boundaries (HALF-OPEN intervals — non-negotiable, document inline):
- `"<1h"`:   `latencySeconds <  3600`
- `"1-6h"`:  `3600  <= latencySeconds <  21600`   (6 * 3600)
- `"6-24h"`: `21600 <= latencySeconds <  86400`   (24 * 3600)
- `">24h"`:  `latencySeconds >= 86400`

A single boundary value (3600s, 21600s, 86400s) falls into the UPPER bucket — verify in unit test.

ALWAYS emit all 4 rows in fixed order even when one or more is empty (e.g. `{bucket:"<1h", tarjeta:0, banco:0, otro:0, total:0}`). The Recharts BarChart in Plan 03-03 needs stable shape across renders; missing buckets cause visual jumps. This also fixes the "Recharts hides zero-bars by default" gotcha flagged in 03-RESEARCH.md.

Implementation: bucket each row via a switch or if-chain on `latencySeconds`, then a switch on `medium` for which counter to increment. O(n) single pass. No floating-point comparisons that could drift (all latency values are integer seconds per Plan 03-01's transform).

**2. `aggregateByDestination(rows: Payout[]): DestinationStats`**

Output type:
```ts
export interface BucketStats {
  count: number;
  montoTotal: number;
  p50Seconds: number;
  p95Seconds: number;
}

export interface DestinationStats {
  tarjeta: BucketStats;
  cuenta_bancaria: BucketStats;
  otro: BucketStats;

```

Three-way partition by `medium`. For each subset, compute count + sum(monto) + p50 + p95 via `quantileSorted` (one sort per subset, reused for both percentiles). Each `BucketStats` is zero-safe: empty subset → `{count:0, montoTotal:0, p50Seconds:0, p95Seconds:0}`.

Note: PAY-04 says "split por destino en TODAS las métricas" — this function is the source of truth for that split. Plan 03-03 KPICards consumes this in addition to `summarizePayouts` (which is the all-mediums total) so the user can see "P50 overall: X, P50 tarjeta: Y, P50 banco: Z".

**3. `aggregateSuccessRate(allRows: Payout[]): SuccessRate`**

Output type:
```ts
export interface SuccessRate {
  /** 0..1 fraction. 0 when totalAttempted === 0. */
  rate: number;
  /** Count where state ∈ COMPLETED_PAYOUT_STATES. */
  completed: number;
  /** Total count (completed + non-completed). */
  totalAttempted: number;
}
```

Special note: this function takes `allRows` (the UN-filtered universe by state — but DOES respect date + empresa from filterPayouts pre-pass). The reason: success rate's denominator MUST include rejected/failed payouts, otherwise the rate is always 100%. Document inline that consumers should call this on `rows.filter(byDateAndEmpresaButNotState)` not on `filterPayouts(rows)` output.

To make this safe, expose a sibling helper:
```ts
/** Filter by date+empresa but keep all states (for success rate denominator). */
export function filterPayoutsByPeriodOnly(rows: Payout[], filters: DashboardFilters): Payout[]
```

This keeps the contract explicit: `summarize/histogram/byDestination` consume `filterPayouts` output (state-filtered); `aggregateSuccessRate` consumes `filterPayoutsByPeriodOnly` output (state-unfiltered).

Compute:
- `totalAttempted = rows.length`
- `completed = rows.filter(r => COMPLETED_PAYOUT_STATES.includes(r.state)).length`
- `rate = totalAttempted > 0 ? completed / totalAttempted : 0` (zero-safe per Plan 02-02 pctDelTotal pattern)
