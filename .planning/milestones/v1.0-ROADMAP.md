# Milestone v1.0: MVP — Tikin Dashboard

**Status:** ✅ SHIPPED 2026-05-06
**Phases:** 1-5
**Total Plans:** 24 (5 phases — 4+4+4+8+4; 1 deferred dentro de Phase 5)

## Overview

Dashboard interno + presentable a clientes corporativos, leyendo en vivo de Google Sheets. Phase 1 montó el esqueleto (auth + adapter + filtros + Modo Presentación) que cada pestaña reutiliza. Phases 2-3 construyeron las pestañas con más data específica (Bonos por revenue, Payouts por diferenciador). Phase 4 cerró Inicio (que agrega de fases anteriores) y Recargas. Phase 5 terminó con Clientes (lista + perfil + "Generar vista para cliente") cerrando el flujo de presentación end-to-end. Único entregable diferido: dominio propio (INFRA-04 / Plan 05-05) — pendiente decisión del usuario sobre qué dominio usar; el dashboard responde en producción en la URL `.vercel.app`.

## Milestone Summary

**Decimal Phases**: None — milestone ejecutó como integer phases 1→5 sin inserciones de emergencia.

**Key Decisions:**

- **Stack Next.js + Sheets API** (vs Looker Studio / Power BI / low-code): mejor presentación a clientes, control total del diseño, margen para crecer. Outcome: ✓ confirmado — design system custom + Modo Presentación + cliente-foco share-URL no se podrían haber hecho en herramienta low-code.
- **Password compartido único** (vs login por usuario): simplicidad para uso interno. Outcome: ✓ confirmado — 5 chars (`T1k1N`) + bcrypt + rate limit 5/5min/IP es suficiente; deuda de seguridad documentada.
- **Lectura en vivo de Sheets** (vs cache / refresh diferido): data siempre fresca. Outcome: ✓ con caveat — `getCachedTransactions()` con React `cache()` deduplica intra-render (DashboardHeader + page comparten 1 fetch); Sheets quota 60/min/SA nunca tocó techo en producción.
- **Sheets como fuente de verdad** (vs integración directa al core): no bloquea valor inmediato. Outcome: ✓ confirmado — adapter abstrae el boundary; el día que se mueva al core de Tikin, solo cambia el adapter.
- **Construcción iterativa por pestaña** (vs especificar todo upfront): visión de pestañas clara, data específica se afina en el camino. Outcome: ✓ — Phase 4 reutilizó shapes de Phase 2 + 3 (Inicio agrega lo que las anteriores ya saben calcular).
- **Vercel CLI 52.0.0 ubicado fuera de PATH** (`/Users/alejoalmeida/.nvm/versions/node/v24.11.0/bin/vercel`): documentado como pre-condición de cualquier deploy task; STATE entries 04-08 + 05-04 lo capturan.
- **PAY-04 reinterpretado como TopBancos** (vs split tarjeta-vs-banco original): la data tiene CERO tarjetas. Honor del espíritu del requirement con granularidad real.
- **PAY-V2-01 (success rate) promovido a v1** durante Phase 3: el campo `BD_Payouts.State` ya existe en producción; el costo era cero.
- **Cliente-foco gate primitive `data-presenter-empresa-hide`**: CSS attribute selector (no React state propagation) — ownership delegada a leaves, NO a la page (consistencia con Modo Presentación). Excepción documentada: page-level wrap cuando un chart degenera bajo empresa filter (e.g. EmpresasActivasChart → y=1 flat line).
- **Per-task atomic commits + plan metadata commit + phase metadata commit**: 88 commits en 10 días = 1 commit por unidad de trabajo verificable; bisect-friendly.
- **Track record de zero-deviation execution**: 12 plans consecutivos con plan-spec literal-block fidelity (02-04, 03-02, 03-03, 03-04, 04-03, 04-01, 04-02, 04-07, 04-08, 05-01, 05-02, 05-03). Page composition + domain library + UI leaf plans demostraron paridad determinista.
